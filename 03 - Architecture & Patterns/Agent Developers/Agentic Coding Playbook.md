# Agentic Coding Playbook: Directing AI to Build AI

**Tags:** #agentic-coding #playbook #team-workflow #ai-cli
**Status:** Active Reference

## 🎭 The Paradigm: Directors, Not Typists
In this vault, human engineers do not write boilerplate. We act as **Technical Directors**. We write strict specifications, provide constrained context, and direct AI Coding Agents (Antigravity CLI, Gemini CLI, Cursor) to generate the actual Python code.

This playbook outlines how teams collaborate to instruct coding agents safely, ensuring the AI generates code that perfectly aligns with our distributed `BaseAgentChassis` architecture.

---

## 🛡️ The 4 Rules of Directing Coding Agents

### 1. Feed the Contract (Context Scoping)
AI coding agents are eager to please and will invent infrastructure if you don't restrict them. 
*   **The Rule:** Whenever you instruct an AI to create a new agent, you **must** load the appropriate skill (`adk-agent-builder` or `adk-infra-builder`) so it knows its boundaries.
*   **The Prompt:** *"You are building a new agent. You must strictly use the methods provided in the BaseAgentChassis framework. Do not write raw Redis, FastAPI, or Postgres connection code."*

### 2. Generate in Layers (The Step-by-Step Workflow)
Do not ask an AI CLI to "Build the Research Agent." It will hallucinate a massive, monolithic file. Direct it layer by layer:
1.  **Layer 1 (Data):** *"Generate the `models.py` file based on this Agent Architecture Spec."* (Review and lock in the Pydantic schemas).
2.  **Layer 2 (Capabilities):** *"Generate the `tools.py` file to fulfill these requirements. Ensure they are standard async Python functions."*
3.  **Layer 3 (The Brain):** *"Generate the `agent.py` file. Use the generated `models.py` and the `@chassis.consume_task` decorator."*

### 3. Test-Driven Defense (TDD)
AI coding agents frequently regress when asked to refactor. We use Test-Driven Development not just for quality, but as a defense mechanism against the AI itself.
*   **The Rule:** Before asking the AI to write complex business logic in `agent.py` or `tools.py`, ask it to write the `pytest` file first.
*   Once the test is locked in, the AI CLI is forced to generate code that passes it.

### 4. Lock Down Integration Points
If Team A is building the Supervisor Agent and Team B is building the Data Agent, the AI CLIs cannot be allowed to hallucinate changes to the API payloads between them.
*   **The Rule:** Pydantic models that define inter-agent communication must be treated as sacred contracts.

## 🤝 The Trilateral Team Workflow

When multiple engineers are using AI to build agents in the same cluster, coordination happens at the **Specification Layer**, not the Code Layer. We divide the team into three distinct roles:

1.  **The Architect (Ray):** Owns the "Universal Core" (`core/` directory) and the global Pydantic contracts. Resolves any disputes and ensures the core remains completely sealed and environment-agnostic.
2.  **The Infrastructure Leads:** Focus exclusively on the `adapters/` directory and deployment manifests (`fleet.yaml`, `docker-compose.yml`). They map the environment dependencies (like Colima or Podman) to the Universal Core using the `adk-infra-builder` skill.
3.  **The Agent Developers:** Focus exclusively on building functional agents (`agent.py`, `tools.py`, `prompts/`). They rely on the Universal Core provided by the Architect and use `mock_infrastructure=True` to build rapidly without waiting for the Infra Leads to finish the adapters. They wield the `adk-agent-builder` skill.

By isolating the AI CLIs into these three distinct zones (`core/`, `adapters/`, and `agents/`), we prevent the AI from hallucinating cross-boundary breaking changes.

## ⚠️ Known AI CLI Anti-Patterns to Watch For
When reviewing code generated by your AI CLI, watch out for these common hallucinations:
*   ❌ Attempting to instantiate `httpx` or `aiohttp` directly instead of using `await chassis.call_agent_sync()`.
*   ❌ Trying to write raw SQL or `psycopg` queries instead of using `await chassis.save_state()`.
*   ❌ Hardcoding model names (`"gemini-1.5-pro"`) instead of relying on `config.yaml`.
*   ❌ Putting massive text prompts inside Python files instead of externalizing them to `.jinja` files.
*   ❌ Infra Leads: AI trying to modify `core/chassis.py` instead of creating a new file in `adapters/`.
