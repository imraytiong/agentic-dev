# 3. Agent Code Reference

This document is for Agent Developers who want to peek "under the hood" at the actual Python code generated by the AI CLI. 

Because we use **Agent-Driven Development**, you shouldn't have to write this code by hand. But understanding the anatomy of an agent will help you debug issues, write better specifications, and understand how the `BaseAgentChassis` is abstracting the heavy lifting.

---

## 📂 The Agent Directory Structure

Every agent in our monorepo (e.g., in `src/agents/sparky/`) follows this exact physical structure:

```text
sparky/
├── agent.py          # The Brain (Routing & Execution)
├── models.py         # The Data Layer (State & Schemas)
├── tools.py          # The Hands (Async Functions)
├── config.yaml       # The Identity (Routing & Settings)
└── prompts/
    └── system.md     # The Personality
```

---

## 🧠 1. The Data Layer (`models.py`)

We use **Pydantic** for all state and data validation. This guarantees that the LLM's output conforms to strict JSON structures before it ever hits your Python logic.

```python
# models.py
from pydantic import BaseModel, Field
from typing import List, Optional

class AgentState(BaseModel):
    """The memory that persists across the agent's execution loop."""
    user_query: str
    current_step: str = Field(default="initializing")
    collected_data: List[str] = Field(default_factory=list)
    is_complete: bool = Field(default=False)
```

**Why it matters:** The `AgentState` is passed into every tool and every execution loop. It is automatically saved to PostgreSQL by the Chassis.

## 🛠️ 2. The Tools Layer (`tools.py`)

Tools are just standard asynchronous Python functions. The Google ADK will automatically convert the docstrings and type hints into the JSON schema required by the LLM.

```python
# tools.py
from google_adk import tool
from .models import AgentState

@tool
async def fetch_user_data(user_id: int, state: AgentState) -> str:
    """
    Fetches user data from the internal API.
    
    Args:
        user_id: The ID of the user to look up.
        state: The current agent state.
    """
    # Mutate the state directly if needed
    state.current_step = "fetching_data"
    
    # Simulate API call
    data = f"Data for user {user_id}"
    state.collected_data.append(data)
    
    return data
```

**Why it matters:** Notice how we pass `state` into the tool. The Chassis automatically injects the current state into the tool at runtime so your tool can read/write memory.

## ⚙️ 3. The Brain Layer (`agent.py`)

This is where the `BaseAgentChassis` shines. Instead of writing boilerplate FastAPI routes or Redis consumers, you just use the `@chassis.consume_task` decorator.

```python
# agent.py
from src.universal_core.chassis import BaseAgentChassis
from .models import AgentState
from . import tools

# 1. Initialize the Chassis (It automatically reads config.yaml)
chassis = BaseAgentChassis()

# 2. Define the execution loop
@chassis.consume_task(queue_name="urgent_tasks", state_model=AgentState)
async def run_sparky(state: AgentState, context: dict):
    """The main execution loop for the agent."""
    
    # Load the externalized prompt
    system_prompt = chassis.load_prompt("prompts/system.md")
    
    # Execute the ADK Agent loop
    result = await chassis.execute_task(
        state=state,
        tools=[tools.fetch_user_data],
        system_instruction=system_prompt,
        user_message=state.user_query
    )
    
    # Mark complete
    state.is_complete = True
    return result

# 3. Expose the FastAPI app for the local Web UI and MCP Server
app = chassis.get_app()
```

**Why it matters:** 
* `@chassis.consume_task` automatically wires this function to FastAPI (for sync UI requests) and Redis (for async background jobs).
* `chassis.execute_task` handles the LiteLLM routing, tool calling loop, and error recovery.

## 🪪 4. The Identity Layer (`config.yaml`)

The chassis needs to know what infrastructure adapters to load and what LLM model to use.

```yaml
# config.yaml
agent:
  name: "Sparky"
  version: "1.0.0"
  
llm:
  provider: "vertex_ai"
  model: "gemini-1.5-pro"
  temperature: 0.2

# The chassis dynamically loads these adapters at runtime!
infrastructure:
  message_queue: "src.infrastructure.public_adapters.standard_redis.RedisAdapter"
  state_store: "src.infrastructure.public_adapters.standard_postgres.PostgresAdapter"
```

**Why it matters:** By changing the strings under `infrastructure`, you can instantly switch this agent from running on your Mac Mini to running on the corporate K3s cluster—without changing a single line of Python code.
